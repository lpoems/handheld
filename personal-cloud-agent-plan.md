# Handheld 方案

## 1. 目标

做一个**自己使用**的云端 agent 系统：

- 通过 `Handheld` 发消息和管理服务
- 只维护**一台长期存在的 CVM**
- 虚拟机内运行完整服务端：`nanobot + 你的扩展服务逻辑`
- 支持多个 `Handheld` 客户端连接同一个服务端
- 支持多个对话复用同一个 `nanobot`
- 所有聊天记录、文件和运行状态都保存在虚拟机磁盘
- 不依赖对象存储作为主持久化
- 需要清空状态时，直接销毁磁盘并重新初始化

目标是把复杂度压到最低，并把日常成本压到接近：

- `token` 成本
- 少量虚拟机按需机器成本
- 少量持久磁盘成本

---

## 2. 总体结论

- `nanobot` 适合作为第一版 agent runtime
- 第一版**不建议做多机、多副本、多持久化后端**
- 第一版采用：
  - **`Handheld` 作为唯一控制端**
  - **始终一台 CVM**
  - **始终一个 `nanobot` runtime**
  - **服务端 = `nanobot` + 你的扩展服务逻辑**
  - **虚拟机磁盘作为唯一主持久化来源**
  - **初始化通过安装脚本自动部署服务端**
- 日常使用时：
  - 不需要每次新建机器
  - 不需要每次关机前上传 OSS / COS
  - 不需要单独维护对象存储恢复链路

---

## 3. 核心架构

系统拆成三层：

### 3.1 Handheld 控制层

负责：

- 接收你在 `Handheld` 内发送的消息
- 用户身份校验
- 展示会话列表和消息历史
- 控制 CVM 初始化、开机、关机、销毁
- 与虚拟机中的服务端通信
- 展示初始化状态、服务状态和任务状态

特点：

- 是唯一入口
- 不承担核心 agent 执行
- 不承担长期状态存储
- 可以有多个 `Handheld` 客户端

### 3.2 虚拟机服务端

虚拟机内运行完整服务端：

- `nanobot`
- 你的扩展服务逻辑
- 会话管理逻辑
- `Handheld` 通信接口
- systemd 服务
- 本地 workspace
- 本地临时文件
- 本地聊天记录与任务产物

这里的服务端不再等同于“裸 `nanobot`”，而是：

**`nanobot` 作为核心 runtime，再加上你为 `Handheld`、会话管理、生命周期控制而补充的服务逻辑。**

### 3.3 本地持久化层

长期状态直接保存在虚拟机磁盘上。

磁盘保存：

- 聊天记录
- 会话索引
- 文件和导出物
- workspace
- 配置文件
- `nanobot` 状态
- 任务日志
- 服务端运行数据

默认不依赖 OSS / COS。

---

## 4. 核心资源模型

### 4.1 固定资源关系

第一版固定采用：

- **多个 `Handheld` 客户端**
- **一台 CVM**
- **一个服务端实例**
- **一个 `nanobot` runtime**
- **多个对话**
- **一个持久磁盘**

也就是：

**多个 `Handheld` 客户端 → 一台 CVM → 一个服务端 → 一个 `nanobot` → 多个对话**

### 4.2 状态保留规则

规则非常直接：

- 只要 CVM 持久磁盘还在，聊天记录和状态就还在
- CVM 关机不会清空数据
- CVM 再次开机后，服务端继续使用原有状态
- 销毁 CVM 磁盘，就等于销毁全部聊天记录和状态
- 销毁后如要继续使用，就重新初始化

### 4.3 为什么这样更适合第一版

这样做可以避免：

- 新建机器后的复杂恢复流程
- 对象存储快照设计
- 关机前强制上传状态
- 多层状态拆分
- 第一版就引入复杂一致性问题

第一版优先目标不是“最强恢复能力”，而是：

- 简单
- 稳定
- 好维护
- `Handheld` 上就能完成全部操作

## 5. 生命周期设计

### 5.1 初始化

初始化是一次显式操作。

当系统尚未创建时：

1. `Handheld` 调用云服务厂商 API 创建 CVM
2. `Handheld` 等待 CVM 可连接
3. `Handheld` 向 CVM 下发执行安装脚本的命令
4. CVM 自动执行安装脚本
5. 安装脚本完成以下工作：
   - 安装依赖
   - 安装部署服务端（ `nanobot` + 扩展服务逻辑形成的代码 ）
   - 创建工作目录与数据目录
   - 写入配置
   - 注册并启动 systemd 服务
6. `Handheld` 轮询初始化状态
7. 初始化完成后，服务端进入可用状态

初始化失败时：

- `Handheld` 能查看日志
- `Handheld` 能重新触发初始化
- 必要时可以销毁 CVM 并重新创建

### 5.2 日常启动

当 CVM 已经存在但处于关机状态时：

1. `Handheld` 调用云 API 开机
2. CVM 启动
3. systemd 自动拉起服务端
4. `Handheld` 检查服务端健康状态
5. 服务可用后开始对话

### 5.3 日常对话

当 CVM 正在运行时：

1. `Handheld` 把消息发送给服务端
2. 服务端把消息路由到对应对话
3. `nanobot` 处理任务
4. 服务端把结果返回给 `Handheld`
5. 服务端把新的聊天记录、文件和状态直接写入本地磁盘

### 5.4 关机

当短期不用时：

1. `Handheld` 请求服务端进入空闲或安全停机状态
2. 服务端完成当前写入
3. `Handheld` 调用云 API 关机
4. CVM 停止运行，但持久磁盘保留

这里不再依赖“关机前上传对象存储”作为主流程。

### 5.5 销毁

当你明确要清空系统时：

1. `Handheld` 调用云 API 销毁 CVM 及其磁盘
2. 所有聊天记录、文件和状态一起删除
3. 后续如要再次使用，就重新走初始化流程

## 6. 服务端设计

### 6.1 服务端定义

服务端不是单独的 `nanobot`，而是：

**`nanobot` + 你的扩展服务逻辑**

扩展部分至少包括：

- `Handheld` 通信接口
- 会话列表管理
- 多对话路由
- 初始化状态上报
- 健康检查
- 日志输出
- systemd 启停管理配合

### 6.2 为什么要直接扩展 `nanobot`

因为要实现这个系统，必然需要额外能力：

- 给 `Handheld` 提供稳定接口
- 控制多个对话
- 管理本地状态
- 与虚拟机生命周期配合
- 支持初始化和健康检查

所以最终形态本来就不是“原封不动的 `nanobot`”，而是：

**以 `nanobot` 为核心 runtime 的定制服务端。**

### 6.3 通信关系

通信关系固定为：

- `Handheld` ↔ CVM 服务端
- 服务端内部 ↔ `nanobot`

而不是：

- `Handheld` 直接面向一个裸 `nanobot` 进程

这样后续更容易补：

- 权限控制
- API 版本演进
- 多客户端支持
- 健康检查与诊断

---

## 7. Handheld 与服务端通信设计

### 7.1 通信方式选择

第一版推荐：

- **`Handheld` ↔ 服务端使用 WebSocket**

原因：

- 支持双向实时通信
- 支持流式返回
- 适合长连接会话
- 适合实时状态更新
- 方便实现“一个字一个字蹦出来”的效果

第一版不优先采用：

- 纯轮询 HTTP
- 仅请求响应式接口

如果后续需要补充普通管理接口，也可以额外提供少量 HTTP API，但主对话链路优先走 WebSocket。

### 7.2 通信目标

通信层至少需要支持：

- `Handheld` 发送用户消息
- 服务端流式返回回复内容
- 服务端推送任务状态
- 服务端推送初始化状态
- 服务端推送错误信息
- `Handheld` 请求会话列表与历史消息
- `Handheld` 请求 CVM 当前服务状态

### 7.3 基本连接模型

建议模型：

1. `Handheld` 建立到服务端的 WebSocket 连接
2. `Handheld` 完成鉴权
3. `Handheld` 订阅当前用户可见的会话与状态事件
4. `Handheld` 发送用户消息或控制指令
5. 服务端返回事件流
6. `Handheld` 按事件类型更新 UI

### 7.4 流式事件模型

建议把所有返回统一抽象成事件，而不是只返回纯文本。

第一版建议至少定义这些事件类型：

- `message_start`
- `message_delta`
- `message_end`
- `task_status`
- `session_state`
- `init_status`
- `server_status`
- `error`

### 7.5 事件语义

- `message_start`
  - 表示一条新回复开始
  - 用于让 `Handheld` 创建一个新的 assistant 消息气泡

- `message_delta`
  - 表示增量文本片段
  - `Handheld` 收到后追加到当前消息内容中
  - 可以形成流式逐字或逐段显示效果

- `message_end`
  - 表示当前回复结束
  - `Handheld` 收到后把该条消息标记为完成

- `task_status`
  - 表示任务阶段变化
  - 例如：`queued`、`running`、`waiting_tool`、`done`

- `session_state`
  - 表示会话元信息变化
  - 例如标题更新、最后消息时间更新

- `init_status`
  - 表示初始化流程进度
  - 例如：`creating_vm`、`booting`、`running_script`、`ready`

- `server_status`
  - 表示服务端健康状态
  - 例如：`starting`、`healthy`、`degraded`

- `error`
  - 表示当前请求或后台任务失败
  - `Handheld` 收到后展示错误信息

### 7.6 最小消息格式

第一版可以统一使用类似下面的结构：

- `type`
- `session_id`
- `message_id`
- `task_id`
- `timestamp`
- `payload`

例如：

- `type = "message_delta"`
- `session_id = "session_123"`
- `message_id = "msg_456"`
- `task_id = "task_789"`
- `payload = { "text": "你好" }`

### 7.7 Handheld 的渲染逻辑

`Handheld` 收到事件后的行为建议如下：

- 收到 `message_start`：创建新消息卡片
- 收到 `message_delta`：把文本追加到当前消息卡片
- 收到 `message_end`：结束流式显示
- 收到 `task_status`：更新顶部状态或会话状态条
- 收到 `init_status`：更新初始化页面进度
- 收到 `error`：显示失败提示

这样就可以自然实现：

- 打字机效果
- 实时任务状态
- 初始化进度提示
- 服务端状态提示

### 7.8 为什么不用只返回完整文本

如果只在最后一次性返回完整文本，会有这些问题：

- 用户等待感更强
- 长回复体验差
- 无法展示中间状态
- 初始化和长任务时没有反馈

所以第一版最好从一开始就把返回设计成流式事件。

## 8. 安装脚本设计

### 8.1 基本思路

安装脚本是初始化的核心。

`Handheld` 不负责亲自部署所有细节，而是：

- 创建 CVM
- 下发“执行安装脚本”的命令
- 查询安装结果

真正的部署逻辑留在虚拟机内执行。

### 8.2 脚本负责内容

安装脚本负责：

- 安装系统依赖
- 安装 `nanobot`
- 部署你的服务端代码
- 创建目录结构
- 写入配置与环境变量
- 注册 systemd 服务
- 启动服务
- 输出成功 / 失败状态

### 8.3 脚本要求

脚本最好满足：

- 可重复执行
- 幂等
- 失败时有明确日志
- 成功后能明确输出 ready 状态

这样即使初始化中断，也可以再次重试，而不是手工修环境。

---

## 9. 腾讯云资源选择建议

### 9.1 第一版推荐组合

推荐组合：

- **`Handheld`**：唯一控制端
- **Tencent CVM**：单虚拟机运行环境
- **持久系统盘 / 数据盘**：唯一主持久化

如果重点是：

- 空闲时尽量降低计算成本
- 保留磁盘状态
- 避免重复初始化

则更推荐：

- **CVM + 保留持久磁盘**

### 9.2 是否还需要对象存储

第一版默认：

- **不需要对象存储作为主链路**

对象存储最多只作为后续可选项：

- 手动备份
- 导出归档
- 灾难恢复副本

而不是主恢复机制。

---

## 10. 第一版默认决策

### 10.1 技术选型

- 控制端：`Handheld`
- 服务端核心：`nanobot`
- 服务端形态：`nanobot + 你的扩展服务逻辑`
- 运行环境：单 `CVM`
- 持久化：单持久磁盘
- 部署方式：安装脚本自动初始化

### 10.2 状态策略

- 不使用常驻数据库
- 不使用对象存储作为主持久化
- 不做多副本状态同步
- 所有主状态直接写入虚拟机磁盘

### 10.3 清理策略

- 保留磁盘 = 保留全部状态
- 关机 = 保留状态，仅停止计算
- 销毁磁盘 = 删除全部聊天记录和状态
- 删除后重新初始化 = 获得一套全新系统

## 11. 测试与验收

### 初始化

- `Handheld` 能成功创建 `CVM`
- `Handheld` 能触发安装脚本执行
- 安装脚本能成功部署服务端
- 初始化完成后服务端处于 ready 状态

### 启停

- 已初始化 `CVM` 能被 `Handheld` 成功开机
- 已运行 `CVM` 能被 `Handheld` 成功关机
- 再次开机后服务端能自动恢复可用

### 对话

- 多个 `Handheld` 客户端都能连接同一个服务端
- 服务端能正确管理多个对话
- `nanobot` 能正常处理消息并返回结果
- 服务端能通过 WebSocket 流式返回消息
- `Handheld` 能正确渲染 `message_start`、`message_delta`、`message_end`

### 持久化

- 关机后重新开机，聊天记录仍然存在
- 关机后重新开机，文件和工作目录仍然存在
- 销毁磁盘后，旧状态完全消失
- 重新初始化后可获得全新状态

### 服务端

- 服务端健康检查可用
- systemd 能自动拉起服务端
- 初始化失败时可以查看日志并重试
- WebSocket 连接稳定可用
- 流式事件格式稳定可解析

---

## 12. 后续可扩展方向

- 给服务端增加更完整的 API 层
- 增加更细的权限控制
- 增加手动备份到对象存储的能力
- 把初始化流程做成预制镜像 + 脚本组合
- 增加运行预算控制：
  - token 限额
  - 最长运行时长
- 给流式协议增加更细的工具事件与调试事件

---

## 13. 一句话版本

第一版方案就是：

**`Handheld` 作为唯一控制端，负责初始化、启停、对话和销毁；云端始终只有一台 `CVM`，里面运行 `nanobot + 你的扩展服务逻辑`；`Handheld` 与服务端通过 WebSocket 做流式通信；所有状态都保存在这台机器的持久磁盘中，保留磁盘就保留全部状态，销毁磁盘就清空全部状态。**
